**Goal:** Bump all dependencies (Go, npm, Docker base, GitHub Actions, tooling) to their current stable versions, then refactor the release CI workflow to use goreleaser as the single source of truth with a test gate, tag validation, dry-run path, and SLSA provenance, while keeping the release draft flag so manual publishing stays a deliberate step.

**Architecture:** Two halves in one plan. Part A bumps dependencies layer by layer with a baseline check after each bump. Part B replaces the hand-crafted three-job release workflow with a single goreleaser-driven job plus a pre-release test gate and tag validation.

**Tech Stack:** Go 1.26, goreleaser v2, GitHub Actions, Docker buildx, golangci-lint v2, Node 20, npm, Tailwind 4, Vitest (absent at plan start, unchanged here), SLSA attestations via `actions/attest-build-provenance`.

## Context

The `2025-01-21-goreleaser-single-source-design` plan unified local builds under goreleaser and is completed. But the CI release path (`.github/workflows/release.yml`) bypasses goreleaser entirely: it runs three hand-crafted jobs (Linux Docker build, Windows native build, assemble release) that duplicate what goreleaser already does locally. Consequences:

- Goreleaser is no longer the single source of truth.
- Any patch to `.goreleaser.yml` does not reach CI.
- No test gate runs before the release builds.
- The release changelog is generated by a bash heredoc in the workflow instead of by goreleaser.
- There is no way to test the release workflow end to end without creating a real (draft) release.
- No SLSA provenance is attached.
- The release is draft (GitHub), which is a user-confirmed safety checkpoint and stays.

On top of this, the project was paused for two months. Go toolchain, Go modules, npm packages, Docker base image, GitHub Actions versions and tooling have drifted. A safe bump pass before the CI refactor gives a clean baseline and avoids fighting an unrelated upgrade while refactoring the workflow.

User directives confirmed:

- Keep `draft: true` in the release action. Manual publish stays.
- Bump everything that can be bumped, in paced layers, baseline-checked after each.
- No GitHub issue or PR interaction during execution (user on standby). The Dependabot PR #48 scope is absorbed here, the PR itself stays untouched.

## Baseline command sets

Two baseline sets exist. Bump tasks (3 to 10) and intermediate verification use the **Quick baseline** only. The **Full baseline** is destructive and runs exactly once at Task 19 as the final end-to-end check.

### Quick baseline (non destructive, fast)

Use after each bump task in Part 2 and after each workflow edit in Part 3.

```
go build ./...
go vet ./...
go test -v ./...
golangci-lint run ./...
npm run lint
npm run css
npm run vendors
```

Expected after a bump:

- `go build` exits 0
- `go vet` exits 0
- `go test` exits 0 (no tests at plan start, so zero pass zero fail is acceptable)
- `golangci-lint` exits 0 or with only acceptable diagnostics that pre-existed
- `npm run lint` exits 0
- `npm run css` produces `web/styles/tailwind.css`
- `npm run vendors` populates `web/scripts/vendors/`

Any failure of a Quick baseline command is a stop. The bump task is rolled back via `git checkout -- <file>` before retrying with a different version.

None of these commands mutates tracked files. The working tree stays clean.

### Full baseline (destructive, end to end)

Use only at Task 19 (Local snapshot verification). This runs goreleaser in snapshot mode via `make all-in-one`, which:

- Calls `before.hooks` that run `compress-game-data.ts --delete-originals`, replacing `web/ao-bin-dumps/*.json` with `.gz`
- Runs `go mod tidy`, which can modify `go.sum`
- Runs the Docker build for Linux
- Restores original data files via `restore-data` on success

```
make all-in-one
```

Expected:

- Exits 0
- Produces exactly 5 files in `dist/` (two binaries, two READMEs, checksums)
- Working tree is clean after the run because `make all-in-one` calls `restore-data` on its own

If `make all-in-one` fails partway (Docker not running, etc.), the working tree may be left dirty. In that case, recover with:

```
git checkout -- go.sum web/ao-bin-dumps/
rm -f web/ao-bin-dumps/*.gz
```

This restores the tracked files and removes the stray gz artifacts.

### Task 12: Add tag validation job

**Files:**
- Modify: `.github/workflows/release.yml`

Tag must point to a commit that is an ancestor of `origin/main`. Otherwise the release is refused.

- [ ] **Step 1: Add the validate job**

Insert a new `validate` job before `test`:
```yaml
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Resolve version input
        id: version
        run: |
          VERSION="${{ github.event.inputs.tag || github.ref_name }}"
          if [ -z "$VERSION" ]; then
            echo "No version available"
            exit 1
          fi
          echo "version=$VERSION" >> "$GITHUB_OUTPUT"
          echo "Resolved version: $VERSION"

      - name: Check tag format
        run: |
          VERSION="${{ steps.version.outputs.version }}"
          if ! echo "$VERSION" | grep -Eq '^[0-9]+\.[0-9]+\.[0-9]+([.-][A-Za-z0-9.]+)?$'; then
            echo "Tag $VERSION does not match X.Y.Z or X.Y.Z-suffix"
            exit 1
          fi

      - name: Ensure tag commit is on main
        run: |
          VERSION="${{ steps.version.outputs.version }}"
          git fetch origin main --depth=0
          if ! git merge-base --is-ancestor "$VERSION" origin/main; then
            echo "Tag $VERSION is not an ancestor of main"
            exit 1

```

```markdown
# Release Workflow

This project uses GoReleaser as the single source of truth for builds, called from a GitHub Actions workflow.

## How to release

1. Make sure main is green in CI.
2. Choose the next version number (semver, `X.Y.Z`).
3. Tag the main branch:
   ```
   git tag -a X.Y.Z -m "Release X.Y.Z"
   git push origin X.Y.Z
   ```
4. The release workflow triggers automatically.
5. Once the workflow completes, a draft release is available on GitHub. Review the artifacts, the release notes, then click Publish.

## Dry run

To test the workflow without creating a real draft release, use `workflow_dispatch`:

1. Go to Actions → Release → Run workflow.
2. Enter a tag (must exist locally or match a real version).
3. Check the "dry_run" box.
4. Click Run workflow.

The pipeline runs the full path (validate, test, build Windows, run goreleaser with `--skip=publish`) without creating or modifying any GitHub release.

## Pipeline stages

1. **validate**: checks tag format and that the tag commit is an ancestor of main.
2. **test**: lints Go + Frontend, runs Go tests and Frontend tests (if Vitest is installed).
3. **build-windows**: builds the Windows CGO binary natively.
4. **release**: runs goreleaser on Linux, builds the Linux binary via Docker, assembles the release, attaches the prebuilt Windows artifact, generates checksums and release notes.
5. Attests SLSA build provenance.
6. Creates a draft GitHub release (not on dry run).

## Manual publish

Releases are always draft. Review the artifacts, the checksums, and the release notes, then publish manually via the GitHub UI or `gh release edit <tag> --draft=false`.

## Troubleshooting

- **validate fails on tag format**: your tag does not match `X.Y.Z[-suffix]`. Delete and retag.
- **validate fails on ancestor check**: your tag commit is not on main. Rebase or retag on a main commit.
- **test fails on Go test**: run `go test -v ./...` locally and fix before retagging.
- **build-windows fails**: check that libpcap headers are available, CGO is enabled, goversioninfo succeeds. The workflow log lists the exact failing step.
- **release step fails on goreleaser**: run `goreleaser check` locally, then `goreleaser release --snapshot --clean --skip=publish` to reproduce.
```
